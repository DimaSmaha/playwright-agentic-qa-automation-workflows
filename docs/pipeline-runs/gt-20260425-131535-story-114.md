## User Story

> **Enable product details navigation from inventory with consistent cart behavior**

Implement a dedicated product details page reachable by clicking a product name or image on the inventory page. The page must show full product information (name, description, price, larger image) and maintain consistent cart behavior: the Add to cart / Remove button must behave identically to the one on inventory, the cart badge must stay accurate, and "Back to products" must return to inventory without losing cart state.

### Acceptance Criteria

| # | Criterion |
|---|---|
| AC1 | Clicking a product name on the inventory page navigates to the product details page |
| AC2 | Product details page displays product name, detailed description, price, and a larger product image |
| AC3 | Product details page has an 'Add to cart' button that changes to 'Remove' when clicked and updates the cart count |
| AC4 | 'Back to products' button returns user to inventory page without losing cart state |
| AC5 | Cart icon is visible and functional on the product details page, allowing navigation to cart at any time |
| AC6 | Cart state (items, count) is synchronized between inventory and product details pages |

## App Exploration Findings

The `gt-story-planner` stage ran a live `playwright-cli` session against `https://www.saucedemo.com` before designing scenarios. Key observations that shaped the test design:

| Finding | Impact on design |
|---|---|
| Cart icon (`data-test="shopping-cart-link"`) has `null` href but is clickable via JS | AC5 scenario confirmed working — no need for href guard |
| Cart badge (`data-test="shopping-cart-badge"`) is **absent from the DOM** when cart is empty; appears only after first add | Scenario 11 (badge not visible) uses `toHaveCount(0)` not `not.toBeVisible()` |
| Product name link pattern: `data-test="item-{id}-title-link"` | `clickProductName()` uses `[data-test$="-title-link"]` + `.filter({ hasText })` |
| Product image link pattern: `data-test="item-{id}-img-link"` | `clickProductImageLink()` navigates into `[data-test="inventory-item"]` to find the sibling link |
| Product detail image: `data-test="item-{slug}-img"` (product-specific) | `assertImageVisible()` uses `[data-test$="-img"]` for product-agnostic match |
| Cart state IS preserved after "Back to products" — backpack shows "Remove" on return | AC4 / AC6 verified in live exploration before spec generation |

## Notable Edge Cases & Observations

1. **Branch name collision** — `task/gt-114` already existed locally from a prior pipeline
   run. `gf-ship` was invoked with `--work-item-id gt-114-specs` to produce `task/gt-114-specs`
   instead of failing. Worth noting: pipeline re-runs on the same story should check for an
   existing branch before invoking `gf-ship`.

2. **Duplicate assertion in scenario 11** — `assertCartBadgeHidden()` appears twice in
   `cart-badge-not-visible-on-details-page-when-cart-is-empty.spec.ts`. This is intentional:
   the skill contract requires a 1:1 mapping between `ideas[]` and `verifications[]`; ideas 2
   and 3 both verify the same badge state from different angles, producing two identical
   assertion calls. Functionally harmless.

3. **`assertCartBadgeHidden()` on non-cart pages** — Both `CartPage` and `ProductDetailPage`
   expose this method. Since all page objects share the same Playwright `Page` instance, calling
   `pages.cart.assertCartBadgeHidden()` correctly checks the badge on whatever page is currently
   loaded (inventory, detail, etc.). This is the correct pattern but may look surprising.

4. **Cart icon visibility** — The AC states "Cart icon must remain visible". In practice the
   `data-test="shopping-cart-link"` element is always in the DOM (confirmed via `querySelector`),
   but the numeric badge (`data-test="shopping-cart-badge"`) only appears when count > 0. Tests
   reflect this distinction: icon navigation is tested after adding an item; badge-hidden tests
   use `toHaveCount(0)` on the badge selector, not on the link.
